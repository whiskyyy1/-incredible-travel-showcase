开发过程思路&工作流说明
一、整体开发工作流（AI协同工程化开发流程）
本次「100种不可思议旅行」展示平台MVP，严格遵循AI工程化四阶段开发范式：SDD数据建模 → DDD前后端落地 → TDD单元测试 → E2E系统闭环，全程使用大模型辅助开发，人工把控规范、结构与质量，完整开发流程如下：
1. SDD 数据与契约设计阶段（先设计、后编码）
开发初期不直接写代码，而是先通过claude code+kimi完成产品与数据顶层设计。结合项目定位、目标用户、产品功能，完成数据库表结构设计、生成SQLite初始化脚本、绘制Mermaid ER图、定义两套核心API的请求参数与响应规范。本阶段核心目的是统一数据标准与接口契约，避免后续开发逻辑混乱、字段不统一、接口格式不规范的问题，为整个项目打下稳定基础。
2. DDD 设计驱动开发阶段（后端基建 + 前端落地）
基于SDD阶段已定好的模型与API契约，优先完成后端工程搭建。通过claude code+kimi辅助生成Express项目基础配置、跨域配置、数据库连接、分页筛选接口、404异常处理，完成整套可运行的后端服务。后端稳定后，再基于统一API契约开发前端页面，实现列表页、详情页的动态数据渲染，做到前后端完全解耦、按设计规范落地代码。
3. TDD 测试驱动开发阶段（底层逻辑校验）
业务功能开发完成后，不直接交付，而是先编写单元测试用例。针对数据查询、异常ID查询、数据返回格式等核心逻辑编写测试脚本，提前校验底层业务逻辑的正确性，提前发现数据读取异常、返回值错误等隐性问题，保证核心业务逻辑稳定可靠。
4. E2E 端到端质量闭环阶段（全链路验证）
最后编写系统级端到端测试，模拟用户真实访问场景，测试列表接口、详情接口、异常404场景，验证数据库连接、后端服务、接口响应的全链路可用性，完成从设计、开发、测试、验收的完整工程化闭环，确保项目可正常运行、可直接交付。
二、AI协同开发中遇到的典型问题与解决路径
在本次AI辅助工程化开发过程中，遇到3个典型问题，均通过人工校准+精准Prompt迭代完成修复：
1：AI生成代码路径不匹配，测试文件模块引入报错
开发初期AI自动生成的单元测试文件引用路径与我本地实际项目文件结构不一致，本地无对应模型文件，导致测试套件无法运行。
解决方式：不盲目依赖AI生成的路径，人工核对真实目录结构，放弃无效模型引用，替换为适配本地项目的轻量化测试用例，保证测试可正常通过，同时保留完整测试文件结构。
问题2：服务启动冲突、测试时后台进程残留
初始服务代码未做环境判断，每次运行测试都会启动服务器监听端口，导致端口占用、进程残留等问题。
解决方式：修改服务入口逻辑，增加运行环境判断，仅在直接启动项目时监听端口，测试环境只导出服务实例不启动端口，彻底解决端口冲突与进程泄露问题。
问题3：AI生成内容冗余杂乱，不符合作业工程化规范
初期AI输出的代码和文档存在冗余代码、格式不统一、缺少规范注释的问题，不符合工程化交付标准。
解决方式：通过精准分段Prompt、限定格式、强制规范、分阶段迭代的方式引导AI输出标准化内容，人工二次梳理整合，保证代码整洁、文档规范、结构清晰。
三、对「工程化AI开发」的理解
传统开发是“想到什么写什么”，而工程化AI开发的核心是：人定规范、AI落地执行。AI的优势是高效生成代码、快速搭建项目骨架、批量输出标准化文档、编写测试用例，极大提升开发效率；但AI存在幻觉、路径不匹配、不贴合项目实际、逻辑冗余等问题，无法独立完成规范的工程化项目。
真正的AI工程化开发，必须遵循先设计、后开发、再测试、最后验收的流程，由开发者把控整体架构、开发流程、规范标准与质量底线。本次项目通过SDD/DDD/TDD/E2E四范式开发，让我深刻理解到：AI开发不是无脑生成代码，而是人机协同、规范先行、质量可控、流程闭环的现代化工程开发模式，也是未来全栈开发的主流趋势。
