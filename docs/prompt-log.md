【SDD阶段】数据建模与API契约生成
====================================================================

我现在在做一个「100种不可思议旅行」内容展示平台的MVP，
该网页的核心主张在于发现那些"你从未想过可以这样旅行"的独特体验，
为追求个性化、差异化旅行方式的用户提供灵感与发现渠道。
目标用户的画像是95后-00后z世代以及反常规生活方式追求者和视觉系内容消费者。

请帮我完成SDD阶段的数据建模和API契约设计：
1. 设计SQLite数据库表结构，包含：
   - 旅行信息表（标题、描述、地点、体验类型、视觉风格、小众程度、故事、贴士、封面图、创建时间）
   - 输出可直接执行的SQL初始化脚本，保存到 sql/init.sql
2. 用Mermaid语法绘制ER图，保存到 docs/er-diagram.md
3. 设计2个核心API接口：
   - GET /api/travels：获取旅行列表，支持按体验类型/小众程度筛选
   - GET /api/travels/:id：获取单条旅行详情
   - 包含请求参数、响应示例、状态码说明，保存到 docs/api-docs.md

【意图说明】
完成项目最核心的数据结构设计，明确前后端交互契约，
为后续所有开发提供统一标准，避免AI开发出现幻觉与偏差。

【DDD阶段】设计驱动开发 — 后端API实现
====================================================================

我现在进入项目的DDD设计驱动开发阶段，
目标是基于已完成的SDD交付物，开发一个可运行的旅行内容展示MVP。

## 项目背景
- 项目名：100种不可思议旅行 - 内容展示平台
- 数据库：SQLite（已完成sql/init.sql，含travels表和测试数据）
- API契约：已在docs/api-docs.md中定义了两个核心接口
- ER图：已在docs/er-diagram.md中完成数据模型设计

## 技术栈要求
- 后端：Node.js + Express，使用原生SQLite驱动操作数据库
- 前端：原生HTML + CSS + JavaScript，不使用框架
- 架构：前后端分离，API与页面完全解耦
- 目录结构：保持现有项目结构，代码放在src/目录下

## 步骤1：开发项目后端API与基础配置
基于已完成的SDD交付物（travels表、API契约），实现以下内容：
1. 创建 package.json，包含 express、sqlite3、cors 依赖
2. 创建 src/server.js，实现：
   - 连接 database/travel.db
   - GET /api/travels 支持筛选、分页、排序，返回 { data, meta } 格式
   - GET /api/travels/:id 返回详情，不存在返回404
   - 启用CORS、错误处理
   - 监听端口3000
3. 代码保持清晰注释
4. 不生成前端，只完成后端与项目配置

【意图说明】
基于SDD输出的契约，完成后端服务骨架与API实现，
让项目具备可运行的后端能力，为前端页面提供数据支撑。
